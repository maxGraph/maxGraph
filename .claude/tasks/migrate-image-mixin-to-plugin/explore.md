# Exploration: Migrate ImageMixin to ImagePlugin

## Scope

Convert `ImageMixin` (and its declaration-merged type augmentation) into a standalone `ImagePlugin`. Move the array property `imageBundles` out of `AbstractGraph` into the plugin (shared-state constraint prevents it from staying on a mixin). Update the single internal consumer (`CellsMixin.postProcessCellStyle`) to go through `graph.getPlugin()`.

This is the proof-of-concept for a larger mixin-to-plugin migration. Keep the blast radius tight.

## Current state (as of this exploration)

### `ImageMixin` source
- `packages/core/src/view/mixin/ImageMixin.ts` — 53 lines. Implements `addImageBundle`, `removeImageBundle`, `getImageFromBundles`. Uses `this.imageBundles` (array living on `AbstractGraph`).
- `packages/core/src/view/mixin/ImageMixin.type.ts` — 36 lines. Declaration-merges the three methods into `interface AbstractGraph`.
- Zero method dependencies on other mixins.

### Storage of `imageBundles`
- Declared in `packages/core/src/view/AbstractGraph.ts:107` as `imageBundles: ImageBundle[] = []`.
- Sits in the block of per-instance complex-type properties (lines 92–125) documented as *"Variables that should be in the mixins but requiring per-instance initialization"* — mixin application caused shared state for object/array types (PRs #751, #879). Moving to a plugin removes that constraint because a plugin is a class instance.

### Mixin registration wiring
- `packages/core/src/view/mixin/_graph-mixins-apply.ts:27,56` — imports and registers `ImageMixin` in the alphabetic list applied to `AbstractGraph.prototype`.
- `packages/core/src/view/mixin/_graph-mixins-types.ts:27` — imports the type-only side-effect.

### Plugin registration wiring (pattern to follow)
- `packages/core/src/view/plugin/FitPlugin.ts` — canonical example: `implements GraphPlugin`, `static readonly pluginId = 'fit'`, `constructor(private readonly graph: AbstractGraph)`, `onDestroy()` no-op.
- `packages/core/src/view/plugin/index.ts` — barrel; `getDefaultPlugins()` returns the array used by `Graph` (not `BaseGraph`). Currently 8 entries.
- `packages/core/src/types.ts:1154-1163` — `BuiltinPluginId` union string literal type. `'fit'` is the only lowercase-short id there; all others are legacy handler names (`CellEditorHandler`, `ConnectionHandler`, ...).
- `packages/core/src/types.ts:1173-1181` — `GraphPluginConstructor` and `GraphPlugin` interfaces. Plugin must expose static `pluginId` and instance method `onDestroy`.

### Internal consumer of `getImageFromBundles`
- **Single call site** in the whole codebase: `packages/core/src/view/mixin/CellsMixin.ts:279` inside `postProcessCellStyle`, which receives a `style.image` key and resolves it against registered bundles.
- `CellsMixin` is a CORE mixin (always applied). Its `PartialGraph` at line 57 currently includes `'getImageFromBundles'`.

### Public API / barrel exports
- `ImageMixin` is **not** re-exported from `packages/core/src/index.ts` — users never import it directly. Only the declaration-merged methods are observable via `AbstractGraph`/`BaseGraph`/`Graph` instances.
- `ImageBundle` class is re-exported at `packages/core/src/index.ts:233` — keep it.

### Tests that touch the current ImageMixin surface
- `packages/core/__tests__/view/BaseGraph.test.ts:326-333` — `test('imageBundles', ...)` asserts that each `BaseGraph` instance gets its own fresh array (per-instance isolation regression check).
- `packages/core/__tests__/serialization/codec/all-graph-classes.test.ts:46` — XML fixture `buildXml()` includes `<Array as="imageBundles" />` as part of the expected serialized form of `<BaseGraph>` / `<Graph>`.
- No dedicated `ImageMixin.test.ts` exists. Coverage report shows the mixin methods are currently uncovered.

### Serialization impact
- `GraphCodec` (`packages/core/src/serialization/codec/GraphCodec.ts`) and `BaseGraphCodec` (`.../BaseGraphCodec.ts`) both extend `ObjectCodec` seeded with a default instance (`new Graph(undefined, undefined, [])` / `new BaseGraph()`). The codec serializes every field present on that seed instance except the ones listed in `excludedFields` (`eventListeners`, `view`, `container`, `cellRenderer`, `selectionModel`, `plugins`).
- Moving `imageBundles` off `AbstractGraph` removes it from the serialized form of `<Graph>` / `<BaseGraph>` XML. Plugin internals are never serialized (`plugins` is already excluded). This is a wire-format breaking change.

### Documentation artifacts
- `packages/core/build/api/*.html` references `addImageBundle` / `removeImageBundle` / `getImageFromBundles` as inherited from `AbstractGraph`. These are generated by TypeDoc from the source — they regenerate from the new plugin types automatically on doc build. No manual edits needed.
- The doc comment in `packages/core/src/view/image/ImageBundle.ts:45` shows `graph.addImageBundle(bundle);` as usage example — must update to the `graph.getPlugin<ImagePlugin>('image')?.addImageBundle(bundle);` form.

### No external consumers in-repo
Searched `packages/website/`, `packages/html/`, `packages/ts-example-selected-features/` and other example packages — no usage of `addImageBundle` / `getImageFromBundles` / `imageBundles`. Nothing to migrate outside `packages/core/`.

## Open design questions resolved during exploration

| Question | Decision | Rationale |
|---|---|---|
| Plugin class name | `ImageBundlePlugin` | Accurate scope (only deals with `ImageBundle` instances), reserves the generic `'image'` namespace for future image-related plugins (background image, warning image from `OverlaysMixin`, etc.) |
| `pluginId` string | `'image-bundle'` | Kebab-case for multi-word ids (consistent with source folder naming); avoids the overly generic `'image'` name. Convention codified in `.claude/rules/architecture/coding-practices.md` → "Plugin ID Naming" |
| Add `'image-bundle'` to `BuiltinPluginId`? | Yes | Plugin ships in `getDefaultPlugins()` ⇒ qualifies as built-in |
| Register in `getDefaultPlugins()`? | Yes | Graph (full-featured) should be a drop-in replacement; BaseGraph users opt in explicitly |
| Keep backwards-compat shim on `AbstractGraph`? | No | Breaking change is assumed (`refactor!`); shim would defeat tree-shaking goal |
| Port existing tests vs add new? | Both — adjust the two impacted tests, add a dedicated `ImageBundlePlugin.test.ts` | No `ImageMixin`-specific test existed; cover the plugin properly |
| Tests for `CellsMixin.postProcessCellStyle` | Add characterization tests in a separate, preceding commit before the refactor touches that method | Method currently has zero tests and the refactor reroutes its internal bundle lookup through `getPlugin()`; pinning behavior first makes the refactor provably safe |
